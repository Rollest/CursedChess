## Цель работы

Разработка веб-приложения на Blazor (.NET) для визуализации работы многоагентной системы на дискретной доске с поиском всех допустимых расстановок агентов, поддержкой пошаговой анимации, учётом стоимости и цвета клеток, фиксацией позиций и сохранением результатов в базе данных.

## Постановка задачи

- Реализовать модель доски N×N, где каждый столбец соответствует агенту.
- Построить алгоритм backtracking, который находит все допустимые конфигурации агентов с учётом активных правил конфликтов.
- Сформировать журнал шагов поиска для последующей анимации на UI.
- Обеспечить задание стоимости и цвета клеток, фиксацию позиций агентов и сортировку решений по стоимости.
- Сохранить конфигурации доски и найденные решения в базе данных (SQLite через EF Core).

## Модель агентов

- Каждый агент отвечает за один столбец доски.
- Состояние агента определяется выбранной клеткой в своём столбце (позиция `Position`).
- Агент может иметь фиксированную позицию (`FixedPosition`) или быть свободным.
- Допустимость позиции определяется набором правил конфликта (`IConflictRule`), которые анализируют текущее состояние среды (уже размещённых агентов).

## Алгоритм поиска (backtracking)

- Агенты обрабатываются по столбцам: для столбца перебираются все строки.
- Для каждой кандидатной позиции:
  - формируется шаг журнала `SearchStep` с типом `TryPlace`;
  - выполняется проверка конфликтов всеми активными правилами (`RowConflictRule`, `DiagonalConflictRule`);
  - при обнаружении конфликта добавляется шаг `ConflictFound`, итерация продолжается;
  - при отсутствии конфликта позиция временно принимается (`Placed`) и рекурсивно обрабатывается следующий столбец.
- При полном заполнении всех столбцов формируется решение `Solution`, вычисляется суммарная стоимость и признак прохождения цветового условия.
- При отсутствии допустимых ходов выполняется откат (`Backtrack`) к предыдущему агенту.

## Архитектура сайта

- **Domain**: сущности `Agent`, `Board`, `Cell`, `Position`, `Solution`, `SearchStep`, `FixedPosition` и интерфейс `IConflictRule`.
- **Application**: сервис поиска решений `ISolutionSearchService`/`SolutionSearchService`, контракты и репозитории (`IBoardRepository`, `ISolutionRepository`).
- **Infrastructure**: `BoardDbContext` (EF Core + SQLite), реализации репозиториев `BoardRepository`, `SolutionRepository`.
- **Presentation**: Blazor-компоненты `Home.razor`, `BoardView.razor`, панель настроек и управления анимацией.

## Структура базы данных

- `Boards` — параметры доски (Id, размер).
- `Cells` — клетки (BoardId, Row, Column, Cost, Color).
- `Agents` — агенты (BoardId, ColumnIndex, признак фиксированности и др.).
- `FixedPositions` — фиксированные позиции агентов.
- `Solutions` — найденные решения (BoardId, TotalCost, IsColorConsistent).
- `SolutionItems` — позиции агентов в решении (SolutionId, AgentId/Column, Row, Column).

## Описание основных классов

- `Board` — описывает доску и содержит коллекции клеток, агентов и зафиксированных позиций.
- `Cell` — хранит координаты, стоимость, цвет и признак занятости.
- `Agent` — описывает агента (столбец, текущую позицию, фиксацию).
- `Solution` и `SolutionItem` — представляют найденное решение и позиции агентов.
- `SearchStep` — элемент журнала поиска, используемый для анимации на UI.
- `IConflictRule`, `RowConflictRule`, `DiagonalConflictRule` — стратегия проверки конфликтов.
- `SolutionSearchService` — реализует backtracking-поиск, формирует список решений и журнал шагов.

## Примеры работы программы

- Пользователь задаёт размер доски, включает/выключает правила конфликта, настраивает параметры стоимости и фильтра по цвету.
- Алгоритм запускается, находит все допустимые конфигурации агентов, формирует журнал шагов.
- На главной странице отображается:
  - визуальная доска с агентами, цветом и стоимостью клеток;
  - панель управления анимацией (шаг вперёд/назад, старт, пауза, сброс);
  - список найденных решений с их стоимостью и признаком прохождения цветового фильтра.
- Решения сохраняются в БД и могут быть повторно загружены и отсортированы.

## Демонстрация многоагентности

- Каждый агент:
  - принимает решение только о своей позиции в столбце;
  - анализирует уже размещённых агентов через набор правил конфликта;
  - может иметь фиксированную позицию или быть свободным;
  - в перспективе может получать собственный набор стратегий `IConflictRule`.
- Таким образом, система моделирует последовательное принятие решений агентами в общей среде (доске), а не просто статическую задачу размещения фигур.

## Вывод

В рамках проекта реализовано Blazor-приложение с многоагентной моделью на дискретной доске, алгоритмом backtracking-поиска всех допустимых расстановок, пошаговой анимацией процесса, учётом стоимости и цвета клеток, фиксацией позиций агентов и сохранением результатов в базе данных. Архитектура разделена на слои Domain, Application, Infrastructure и Presentation, что упрощает расширение системы новыми правилами конфликтов и дополнительными сценариями использования.
